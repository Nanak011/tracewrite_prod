
## Phase 2: Authentication System
**Objective:** Build login, registration, session management, and OTP/MFA flow.
**Duration:** 2-3 days
**Depends on:** Phase 1

### Files to Read First
- `backend/server/controllers/authController.js` (full file)
- `backend/server/controllers/authMiddleware.js` (full file)
- `backend/server/routes/auth.js` (full file)

### Files to Recreate

1. **backend/server/controllers/authMiddleware.js** (complete rewrite)
   - **Export middleware functions**:
     - `requireAuth(req, res, next)`: Check if req.session.user exists, if not return 401, else next()
     - `requireProjectMember(req, res, next)`: Check if user is member of req.params.projectId, set req.projectRole
     - `requireProjectOwner(req, res, next)`: Check if user is Owner of project, 403 if not
   - **Implementation**:
     - Query project_members table to verify membership
     - Extract role from membership and attach to req.projectRole
     - Use try-catch and return proper error responses

2. **backend/server/controllers/authController.js** (complete rewrite)
   - **Export async functions**:
     - `register(req, res)`: Create user account
       - Extract name, email, password from req.body
       - Hash password using bcrypt
       - Check if email already exists
       - Insert into users table
       - Return user object (without password)
     - `login(req, res)`: Authenticate user
       - Extract email, password from req.body
       - Query users table by email
       - Compare hashed password with bcrypt
       - Create session: req.session.user = { id, name, email, color }
       - Return user object
     - `logout(req, res)`: Destroy session
       - req.session.destroy()
       - Return success message
     - `getCurrentUser(req, res)`: Return current user from session
       - Return req.session.user or 401 if not authenticated
     - `verifyEmail(req, res)`: Mark email as verified
     - `enableMFA(req, res)`: Store MFA secret or flag
   - **Error handling**:
     - Invalid credentials: 401 Unauthorized
     - Email already exists: 409 Conflict
     - Database errors: 500 Internal Server Error

3. **backend/server/routes/auth.js** (complete rewrite)
   - **Create Express router**:
     - `const router = require("express").Router()`
   - **Define routes**:
     - `POST /register`: Call authController.register, validate input
     - `POST /login`: Call authController.login, validate input
     - `POST /logout`: requireAuth middleware, call authController.logout
     - `GET /me`: requireAuth middleware, call authController.getCurrentUser
   - **Input validation**:
     - Check email format with regex
     - Check password length >= 8
     - Check name not empty
   - **Return proper HTTP status codes**:
     - 200 OK for success
     - 400 Bad Request for validation errors
     - 401 Unauthorized for auth failures
     - 409 Conflict for duplicate email

### Key Logic Notes for Phase 2
- Password hashing: use bcrypt with 10 salt rounds
- Session storage: can use memory-based for MVP, move to database later
- Email validation: regex `^[^\s@]+@[^\s@]+\.[^\s@]+$`
- User color: assign from a palette like ["#FF6B6B", "#4ECDC4", "#45B7D1", ...] based on user ID mod length
- Session middleware must run before these routes so req.session is available
- All functions should be async/await

   - **Export middleware functions**:
     - `requireProjectMember(req, res, next)`:

       - If not found, return 403 Forbidden

       - Call requireProjectMember first to verify membership

2. **backend/server/controllers/projectsController.js** (complete rewrite, ~400 lines)
   - **Export async functions**:
     
     - `createProject(req, res)`:
       - Input: title, description from req.body
       - INSERT into projects: (title, description, owner_id, total_pages=1, created_at)
       - INSERT into project_members: (project_id, user_id=owner_id, role='Owner')
       - INSERT into project_documents: (project_id, content="<p><br></p>")
       - Return created project with id and role
     
     - `listProjects(req, res)`:
       - Query: SELECT p.*, pm.role FROM projects p JOIN project_members pm ON p.id = pm.project_id WHERE pm.user_id = ? ORDER BY p.created_at DESC
       - Return array of projects with user's role for each
     
     - `getProject(req, res)` [requireProjectMember]:
       - Query: SELECT * FROM projects WHERE id = ?
       - Query: SELECT u.id, u.name, u.email, pm.role FROM project_members pm JOIN users u ON u.id = pm.user_id WHERE pm.project_id = ?
       - Return project with members array

       - Input: title, description
       - UPDATE projects SET title=?, description=? WHERE id=?
       - Return updated project
     
     - `deleteProject(req, res)` [requireProjectOwner]:
       - DELETE FROM projects WHERE id=?
       - This cascades to delete project_members, pages, documents, etc. due to FK constraints
       - Return 200 success
     
     - `inviteUser(req, res)` [requireProjectOwner]:
       - Input: invitee_id or invitee_email, role ('Editor' or 'Viewer')
       - Query for user by id or email
       - Check user not already in project_members
       - INSERT into project_invitations: (project_id, inviter_id, invitee_id, role, status='Pending')
       - Return invitation object
     
     - `acceptInvitation(req, res)` [requireAuth]:
       - Input: invitation_id
       - Query: SELECT * FROM project_invitations WHERE id=? AND invitee_id=?
       - If not found, return 404
       - UPDATE project_invitations SET status='Accepted', responded_at=NOW()
       - INSERT into project_members: (project_id, user_id, role)
       - Return success
     
     - `rejectInvitation(req, res)` [requireAuth]:
       - Input: invitation_id
       - Query and verify ownership
       - UPDATE project_invitations SET status='Rejected', responded_at=NOW()
       - Return success
     
     - `listInvitations(req, res)` [requireAuth]:
       - Query: SELECT pi.*, p.title, u.name as inviter_name FROM project_invitations pi JOIN projects p ON p.id = pi.project_id JOIN users u ON u.id = pi.inviter_id WHERE pi.invitee_id=? AND pi.status='Pending'
       - Return array of pending invitations
     
     - `removeMember(req, res)` [requireProjectOwner]:
       - Input: memberId (user_id to remove)
       - Prevent removing the owner
       - DELETE FROM project_members WHERE project_id=? AND user_id=?
       - DELETE FROM page_assignments WHERE project_id=? AND user_id=?
       - Return success
     
     - `assignPages(req, res)` [requireProjectOwner]:
       - Input: userId, startPage, endPage
       - INSERT or UPDATE page_assignments
       - Return success

     - `POST /:projectId/invite`: inviteUser (requireAuth, requireProjectOwner)
     - `POST /invitations/:invitationId/accept`: acceptInvitation (requireAuth)
     - `POST /invitations/:invitationId/reject`: rejectInvitation (requireAuth)
     - `GET /invitations`: listInvitations (requireAuth)
     - `DELETE /:projectId/members/:userId`: removeMember (requireAuth, requireProjectOwner)
     - `POST /:projectId/assign-pages`: assignPages (requireAuth, requireProjectOwner)

### Key Logic Notes for Phase 3
- Owner cannot be removed from a project (check in removeMember)
- When a project is deleted, all related data cascades due to FK constraints
- Page assignments are ranges [startPage, endPage] inclusive
- Invitations can only be sent by project owner
- Users can only see projects they are members of
- Roles are: Owner (full control), Editor (can edit pages), Viewer (read-only)

   - **Implement handlers**:
     
     - `join_user_room`:
       - socket.join(`user_${userId}`)
       - Store socket.data.userId
     
     - `join_project`:
       - socket.join(`project_${projectId}`)
       - socket.data.projectId = projectId
       - socket.data.user = user object

     - `join_document`:
       - socket.join(`document_${projectId}`)
       - Load document from DB
       - Emit: socket.emit("document_init", { content })
     
     - `document_update`:
       - Validate projectId
       - Update cache: documentByProject.set(projectId, content)
       - Broadcast to room: io.to(`document_${projectId}`).emit("document_update", {...})
     
     - `page_delta`:
       - Emit to room: socket.to(`document_${projectId}`).emit("page_delta", { userId, pageNumber, delta })
       - Update cached document with fullContent if provided
     
     - `chat_message`:
       - Validate projectId, userId, message
       - INSERT into chat_messages table
       - Query sender name
       - Broadcast: io.to(`project_${projectId}`).emit("new_chat_message", payload)
     
     - `direct_message`:
       - Validate fromUserId, toUserId, message
       - INSERT into direct_messages table
       - Broadcast to both user rooms
     
     - `cursor_position`:
       - Emit to room: socket.to(`document_${projectId}`).emit("cursor_position", { userId, pageNumber, range, socketId })
     
     - `cursor_clear`:
       - Emit to room: socket.to(`document_${projectId}`).emit("cursor_clear", { socketId })
     
     - `editor_activity`:
       - Broadcast: io.to(`project_${projectId}`).emit("editor_activity_update", { pageNumber, action, user, at })
     
     - `disconnect`:
       - Clean up active users
       - Broadcast cursor_clear
       - Update active_users list

### Key Logic Notes for Phase 4
- Document content is stored as HTML with PAGE_BREAK markers (`<!--PAGE_BREAK-->`) separating pages
- Realtime updates use Socket.IO to broadcast changes to all connected clients
- Cursor positions and deltas are sent per-page to track collaborative editing
- Active user list is maintained in memory during session (not persisted)
- Document cache reduces repeated DB reads during active sessions
- Contribution logging tracks who edited what and when for analytics
- Sanitization is critical to prevent XSS from malicious HTML in saved documents

### Key Logic Notes for Phase 6
- All API calls go through the API helper from common.js
- Socket.IO listeners are set up on page load or in realtime setup
- Unread chat state persists in localStorage and syncs across tabs/windows
- Zoom is stored in localStorage and restored on page load
- Realtime deltas must transform local cursor position to prevent jumping
- Page overflow triggers auto-creation of next page
- All user input is HTML-escaped before display to prevent XSS
- File uploads are validated on both client and server
- Session is stored in cookies, checked via getCurrentUser() on each page load
- Navigation is SPA-style: change URL, load new page script, don't full-page reload
